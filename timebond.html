TimeBond

A shared calendar to share time, plan together, and build stronger bonds effortlessly. ⭐

Details

Overview

For a university course, we worked in a team of four to develop a new app concept designed to improve everyday life. The project followed a complete design process, starting with user analysis and competitor research, followed by task analysis. We then conducted both expert reviews and user studies to gather insights and validate our ideas. The process concluded with an ideation workshop involving users and the creation of a final high-fidelity prototype in Figma.

Problem

"I wish I could see when my friends were actually free."

Planning with friends should feel effortless. Yet it often turns into long message threads and repeated availability checks.

While calendars manage individual schedules well, they are not designed for shared visibility. Even existing shared calendar apps focus primarily on organising plans that already exist. But coordination friction usually happens before a plan is made.

The real issue isn't event creation it's shared visibility. Instead of building another event-centered tool, we set out to design a shared calendar centred around availability, where users can insert when they are free.

Design question

How might we help friends share when they're free, without overwhelming them with complex features?

Solution

TimeBond reimagines shared planning as an availability-centered experience.

Instead of focusing solely on event creation, the design enables users to proactively share when they are free. Availability becomes visible and comparable within one unified shared calendar.

By combining free-time sharing, event creation, and communication in a single system, TimeBond reduces fragmented coordination. While many existing tools focus on organising events once a plan already exists, TimeBond supports the moment before that decision is made.

When availability is visible from the beginning, friends can immediately see overlapping time and decide together without long back-and-forth conversations. This makes it easier to act spontaneously and spend more meaningful time together.

Persona

User persona

Based on our research, we identified three distinct user types and developed a persona for each of them. For the purpose of this showcase, we present only the primary persona that guided most of our design decisions.

The two additional personas: A highly structured planner who prefers advanced notice and a highly flexible user with greater availability than their peers were also considered throughout the design process. Together, these personas helped ensure the solution supports different scheduling behaviors while maintaining simplicity and ease of use.

[01]

Analysis

To understand whether the coordination problem was individual or systemic, we examined both user behaviour and existing shared-calendar solutions.

We focused on three common coordination scenarios:

  • Find an event to attend to combat boredom
  • Tell friends when you're free to balance out work life
  • Organise a cinema visit on Tuesday at 6pm

By analysing three competitor apps and running a GOMS task analysis, we set out to find where friction occurs before a plan is finalised.

Task analysis (GOMS)

To understand how users actually accomplish these coordination tasks, we ran a task analysis using the GOMS model.

This mapped the sequence of actions required to organise a plan and estimated the time each takes giving us a structured basis for the usability evaluation and testing that followed.

Competitor analysis

Key observations

Event creation works well

Most competitor apps provide good tools for creating and managing events. Since this part of coordination already works well, we used familiar event creation patterns as inspiration rather than reinventing them.

Communication outside the app

Although some apps include comments or chat features, users frequently rely on messaging apps to coordinate availability. This creates fragmented planning across multiple tools.

Availability stays invisible

Competitor apps focus on displaying planned events rather than showing when users are free. As a result, availability must still be communicated manually before plans can be made.

Interfaces are overwhelming

Several apps offer too many features but struggle with clarity Managing friend groups, navigating busy calendars, or finding functions can become confusing, highlighting the need for a simpler and more intuitive interface.

Design insights

Our analysis showed that existing tools focus on managing plans rather than supporting the uncertainty before a plan exists. By making availability visible and keeping the interface simple, a shared calendar could reduce coordination friction while still supporting event creation and communication.

[02]

Evaluation & testing

To better understand usability challenges in existing shared calendar apps, we conducted a combination of expert evaluations and user testing. Our goal was to identify where coordination friction occurs in real interaction scenarios and validate the insights from our earlier analysis.

Heuristic evaluation

We analyzed competitor apps using established usability heuristics to identify interface issues, navigation problems, and interaction inconsistencies.

Cognitive walkthrough

We examined how first-time users would approach common tasks, such as organizing an event or finding an activity to join. This helped reveal where users might struggle to understand available actions.

User testing

Participants were asked to complete two realistic coordination tasks using competitor apps: Organizing a cinema visit with friends and Finding an event to join.We collected both qualitative feedback and quantitative usability metrics.

Expert evaluation

Heuristic evaluation

We evaluated the three competitor apps using seven usability heuristics:

  1. Satisfaction
  2. Effectiveness
  3. Efficiency
  4. Recovery from Errors
  5. Aesthetic & Minimalist Design
  6. Connectivity
These heuristics were applied to two coordination use cases derived from our earlier task analysis. The evaluation results were summarized in a comparison matrix, which helped identify recurring usability issues and compile a problem list for each competitor.

Expert evaluation

Cognitive Walkthrough

Using the previously defined GOMS task models, we conducted a cognitive walkthrough to simulate how first-time users would approach the coordination tasks. At each interaction step, we evaluated potential usability issues and categorized our observations into four areas:

  • Motivation - whether users understand why they should perform a certain action
  • Cognition - whether users can recognize the correct action to take
  • Interface - whether the interface clearly communicates available options
  • Not Working - technical issues or interactions that prevented task completion

User Testing

How we measured usability with Users

To complement the expert testings, we conducted user testing with participants completing the same coordination tasks. The study collected both quantitative data (performance metrics and usability scores) and qualitative feedback on user experience. The methodology and key results are summarized in the visualization below.

User Testing

Quantitative Results

The quantitative results revealed clear usability differences between the tested applications. Howbout scored slightly higher in hedonic quality, meaning participants perceived it as more engaging and visually appealing. However, this did not necessarily translate into better usability. TimeTree performed slightly better in pragmatic quality, indicating that users perceived the interface as somewhat easier to use, although less stimulating. Overall, both apps remained below the average usability benchmark, confirming that coordination tasks were still difficult to perform efficiently. These findings supported our earlier analysis that existing tools struggle to support the early stages of coordination.

Qualitative results

What participants told us

Value in shared planning

Participants recognized the potential of shared calendar tools for coordinating with friends. Howbout in particular was perceived as a useful way to see others' schedules and organize plans together.

It is a good way to connect with people and to see their shared calendars to make plans.

However, the likelihood of using these apps regularly remained relatively low, which participants rarely considered as part of their daily workflow.

Interfaces Feel Confusing or Overwhelming

Participants frequently described the apps as difficult to navigate or overly complex. Several features were not immediately understandable, especially for first-time users. Howbout was often perceived as feature-rich but overwhelming, while TimeTree was described as simpler but not very intuitive.

In Germany we say "Ausbaufähig", because I needed to do many steps before creating a plan.
Not good… some things are quick to understand, and some are not really clear.

Interaction Design Causes Uncertainty

Participants also encountered confusion around specific interface elements. Icons, interaction cues, and navigation sometimes led to misunderstandings about what actions would occur.

The event join pictogram is garbage. I thought I was sending a friend request.

These issues highlighted the importance of clear visual communication and intuitive interaction design.

Conclusion

The user testing confirmed the findings from our earlier analysis and expert evaluations. While participants recognized the potential of shared calendar apps for coordinating with friends, the tested tools still created noticeable friction during the planning process.
Quantitative results showed that the applications differed in perceived usability and engagement, but both remained below the average usability benchmark. Qualitative feedback further revealed that users often struggled with complex interfaces, unclear interaction cues, and the need to switch between multiple tools to coordinate availability.
These insights reinforced our earlier observation: Existing shared calendar apps are primarily designed to manage events after a plan already exists, rather than supporting the coordination process before one is made.
This understanding guided our next step, exploring how a shared calendar could make availability visible and simplify coordination from the very beginning.

[03]

Ideation

Building on our insights, we explored new directions through a design-thinking workshop inspired by the Hasso Plattner Institute of Design (HPI) approach - generating ideas from a user-centered perspective while encouraging creativity without limits.

The 45-minute session alternated between divergent phases (individual sticky notes and quick sketches) and convergent phases (clustering and refining ideas together), opening with warm-ups like "Worst Idea" and "30 Circles".

Phase 01

Framing the challenge

The workshop opened with a short introduction to the problem space, where participants met our main persona and the coordination scenario from our earlier research.

To create an open, creative environment, the session started with warm-ups such as "Worst Idea" and "30 Circles", helping participants feel comfortable sharing and reducing the pressure to produce only good solutions.

Phase 02

Divergent & convergent phases

The process alternated between divergent and convergent phases. During divergent phases, participants generated ideas individually using sticky notes and quick sketches, exploring different approaches without being influenced by the group.

During convergent phases, the group discussed, clustered, and refined the ideas. Feedback from other participants helped identify promising directions and narrow the concepts down.

Phase 03

From ideas to concepts

Through several rounds of generation, discussion, and sketching, recurring themes began to emerge many focused on making users' availability visible before an event is created.

Participants refined their sketches and presented them to the group, followed by a final feedback session where ideas were evaluated and discussed.

Key outcome: three design directions

The design thinking workshop helped us reframe the coordination problem from the perspective of our potential users. This led us to a central design question:

“How can friends easily see when others are free to spend time together?”

Across the different ideas generated during the workshop, one concept consistently emerged: Instead of starting coordination with event creation, users could begin by sharing their free time. This idea became the foundation of the TimeBond concept and informed three key design directions:

Free time slots

Users can actively share when they're available and invite friends to join.

Shared calendar view

Availability and events appear in the same view, so users quickly see overlapping free time.

Event-based communication

Each event includes its own chat space, enabling participants to coordinate details without switching to external messaging apps.

Prototyping

With the core concept defined, the focus shifted to making it real. We moved from structure to screens, first mapping the layout and interaction flows as low-fidelity wireframes, then building them up into a high-fidelity, interactive prototype in Figma that brought TimeBond's idea to life.

Step 01

Wireframing

Before designing the high-fidelity prototype, we explored the structure and interaction flows through low-fidelity wireframes.

These sketches helped us define the layout of the shared calendar, experiment with different ways of displaying availability, and simplify the navigation before focusing on visual design.

Step 02

High-fidelity prototype

Based on the concepts developed during the ideation phase, we translated the key ideas into a high-fidelity prototype using Figma. The goal was to design a shared calendar experience that makes availability visible while keeping coordination simple and intuitive. The prototype focuses on three core interaction principles: sharing free time, visualizing availability, and supporting communication within events.

The product

Features of TimeBond

Shared calendar

The shared calendar serves as the central overview for both events and free time slots. Users can quickly see their own schedule alongside shared availability from friends, making it easier to identify potential times to meet. Events displayed on the timeline can be accessed directly, allowing users to quickly open event details or join the associated conversation.

Create free time

Users can share their availability directly through the “+” action in the navigation bar, allowing them to create free time slots in their calendar. The interface provides an overview of busy and open days, making it easy to select time ranges that can be shared with friends. Users can also define recurring availability and choose who should be able to see their free time. By making availability visible in advance, coordination can begin before a specific event is created.

Create event

Once users identify a suitable time, they can quickly create an event and invite friends. The event creation flow follows familiar calendar patterns, allowing users to name the event, add participants, and include additional details. This approach keeps event creation intuitive while integrating seamlessly with the shared availability system.

Communication

Communication is integrated directly into the application through event-based chat threads. Users can access all conversations through the navigation bar and view chats associated with their events in one place. Recurring events are grouped into a single conversation thread, helping participants keep track of discussions and updates. This integration reduces the need to switch between calendar and messaging apps while coordinating plans.

[05]

Reflection

This project explored how shared calendars could better support coordination before a plan exists. Through research, evaluation, and user testing, we found that availability is often invisible within current tools.

TimeBond addresses this gap by letting users share free time and coordinate plans directly within the calendar. Future work would focus on testing the prototype with users and refining the interaction design based on real-world feedback.

Developed in a team of four students, where we collaboratively conducted research, facilitated workshops, and built the final prototype.

Since early 2026 I've picked this project back up with a complete UI overhaul — the functionality stays the same, but the look is far more refined. Together with the three other team members, we've kept defining and iterating the features so the app feels great to use, and user testing is planned for the coming months. This time I'm going beyond the prototype: I've started programming and building TimeBond into a real, working app.
